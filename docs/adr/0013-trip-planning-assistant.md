# ADR-0013: Trip Planning Assistant

## Status

Accepted

## Context

Feature #4 in `docs/ai-integrations.md`: after a Booking is confirmed, a hirer-only
"Plan my trip" panel lets the Hirer enter free-text interests and receive a
day-by-day **Itinerary**, generated by Claude and **grounded in retrieved local
knowledge** (attractions, campground FAQs, area policies). It is the repo's RAG +
background-jobs + context-management exercise.

Pipeline: `interests + region + dates → retrieve region chunks → RAG prompt →
structured itinerary`, run asynchronously (Sidekiq) with the frontend polling for
the result. Backend service `Ai::TripPlanner` (subclass of `Ai::BaseAiService`, per
ADR-0009); retrieval reuses the Ollama + pgvector + `has_neighbors` embedding stack
from ADR-0011.

The design was settled in a grill session. The decisions below record the tree we
walked and *why* — several cut against the written spec or the obvious answer, so
they need context to stop a future reader from "fixing" them.

## Decision

### Knowledge corpus is LLM-generated and synthetic, not hand-authored

The `app/knowledge/` corpus is **written once by Claude** (a `knowledge:generate`
rake task), committed, then treated as static content. It is *not* researched,
real-world data. This is a deliberate scope call: the learning goal is the **RAG
pipeline** (chunk → embed → retrieve → ground), not curating a travel database, and
the pipeline is identical whether the source text is authored by hand or by a model.

The direct consequence — **the itinerary contains invented facts** — is contained by
two other decisions below: the output carries an explicit `disclaimer`
("suggestions only, verify locally") and the feature is **coverage-gated** so we
never generate a plan we can't ground at all. A future reader tempted to present
itineraries as authoritative should read this paragraph first.

### Real vector retrieval, not whole-doc injection

Because each synthetic region doc is small, we *could* skip embeddings and stuff the
whole file into the prompt. Rejected: that hollows out the headline RAG lesson. We
build genuine retrieval — a new **`KnowledgeChunk`** model (pgvector, `has_neighbors`,
768-dim, same space as `ListingEmbedding`). At query time we embed the query, filter
to the Booking's Region, and take the **top ~6** nearest chunks into the prompt.

### Chunking: one chunk per H2 section

Each Region markdown file is split on `##` headings; each section (e.g. "Beaches",
"Campground FAQs", "Wet-weather options") becomes one `KnowledgeChunk` with its
heading stored. Semantically coherent, human-readable, and a per-chunk `content_hash`
(mirroring ADR-0011) means editing one section re-embeds only that chunk. Fixed-size
token windows with overlap (the textbook technique) were rejected as poor fit for a
small hand-shaped corpus — they split mid-thought and add a tokenizer step for no
gain here. Embedding is a deploy-time `knowledge:embed` rake task, idempotent on
`content_hash`; never per-request.

### Region is sub-state, resolved via a `region` column set by a canonical resolver

The unit of grounding is a **sub-state Region** (e.g. "Great Ocean Road"), not the
whole state — finer regions yield materially better itineraries. A Listing resolves
to at most one Region:

- **`region` (slug string) is a real column on `rv_listings`**, set in a
  `before_validation` by a single `Region::Resolver.call(town:, postcode:, state:)`.
  Retrieval reads `booking.rv_listing.region` directly — no query-time string
  matching off to the side (the "operational cleverness" the repo's conventions
  avoid). Requires a migration + backfill of existing rows.
- **The Region vocabulary lives in a config manifest** (`app/knowledge/regions.yml`:
  slug, name, state, town/postcode match rules, doc filename), **not a DB table**.
  Regions are static reference data defined at deploy time and never CRUD'd at
  runtime, so a table would be a CRUD-less join target that must be seeded and kept
  in sync with the markdown files. The manifest is the single source of truth for the
  resolver, the coverage check, and chunk loading. `KnowledgeChunk` carries the
  region slug as a string.

### Coverage gating: only grounded plans, ever

Trip planning is offered **only when the Booking's Region has a corpus**
(`confirmed? && region&.has_corpus?`). Uncovered Bookings (region resolves to `nil`,
or a Region with no chunks) hide the panel; a direct `POST` against one returns
**422**. This keeps every itinerary we ever generate grounded — no pure-hallucination
plans — which is what makes the synthetic corpus acceptable. Rejected alternatives: an
always-on ungrounded fallback (defeats the honesty containment above), and a
state-level corpus fallback (would require authoring *both* region and state docs,
partly undoing the sub-state decision).

### One regenerable `TripPlan` per Booking

`trip_plans` has a **unique index on `booking_id`**. Status machine:
`pending → processing → completed | failed`, with `interests`, `itinerary` (jsonb),
and `error` columns. Re-running resets the same row to `pending` and re-enqueues
(overwrite). The poller reads `GET .../trip_plan`. An append-only history of
generations was rejected as scope the MVP doesn't need — it adds "which is current"
bookkeeping and a busier UI. The unique index also makes double-submit idempotent.

### Output: structured day-by-day JSON keyed to real dates

`Ai::TripPlanner`'s `output_schema` is fully structured:
`{ summary, disclaimer, days: [{ date, title, segments: [{ part_of_day, activity,
detail }] }] }`. Days map to the Booking's real calendar dates. This is the
structured-outputs lesson done properly (validatable, richly renderable) and gives
the `disclaimer` field a home. A single markdown blob was rejected — no per-day
structure, harder to style or use programmatically.

### Context management: cap planned days, raise the token ceiling

Days map one-to-one to Booking nights, so a long stay means large retrieval *and*
large output. We **plan the first `MAX_PLANNED_DAYS` (7, configurable)** and, for
longer Bookings, add a closing note ("plus N more nights to explore"). This bounds
retrieval size, output tokens, latency, and cost predictably — the
context-management goal made concrete. `BaseAiService`'s hardcoded `max_tokens: 1024`
is made **overridable via a `MAX_TOKENS` constant** (default preserved);
`TripPlanner` raises it to a fixed ceiling sized for N days. Model stays
`claude-sonnet-4-6` (adequate, cost-conscious).

### Interests optional; retrieval query blends Booking context

Interests are **optional**. The retrieval query is
`[interests, region.name, season(start_date), party_size].compact.join`. Blank
interests still retrieves the Region's top chunks (query = region + season) and
yields a solid default plan; adding interests refines it. Season is derived from the
`start_date` month (southern hemisphere). This gives value at zero effort, better
with input — rejected the alternative of requiring a minimum-length interests string
(no zero-effort plan, extra empty state).

### Three cost guards on `POST .../trip_plan`

Trip-plan generation is the most expensive AI call in the app and is
hirer-triggered and regenerable, so all three reusable guards apply:

1. **In-flight lock** — reject `POST` (**409**) while status is `pending/processing`
   (natural on the unique row).
2. **Per-user rate limit** — the ADR-0010 `rate_limit` macro, its **own bucket** (not
   a global AI budget), small N/hour.
3. **Unchanged no-op** — when a `completed` plan exists and `interests` (+ region) are
   unchanged (a `content_hash`, mirroring ADR-0011), return the existing plan without
   enqueuing a paid call.

### Job failure: no auto-retry, manual re-run

`GenerateTripPlanJob` sets **`sidekiq_options retry: 0`**. Sidekiq's default 25×
backoff would mean up to 25 *paid* Claude calls per Booking and a plan stuck
`processing` for hours — flatly against the cost guards. On any `Ai::Error` the job
marks the `TripPlan` `failed` and stores the message (the `BaseAiService` `ensure`
block has already logged the `ai_requests` row). The frontend shows the failure with
a "Try again" button that re-`POST`s through the guards. Bounded transient retry
(`retry: 2` on `ApiError` only) was the runner-up; deferred to keep v1 predictable.

### Booking-detail page is a prerequisite (not in the spec)

The spec places the panel "on the booking detail page" — **which does not exist**.
Today `BookingPage` is the creation flow and `BookingsPage` is the list; bookings
have no `show` route. So this feature ships a prerequisite: **`GET
/api/v1/bookings/:id` (show) + a new `BookingDetailPage`**, reached from
`BookingsPage`, where the collapsible "Plan my trip" panel lives. The show payload
includes `trip_planning_available`. The `trip_plan` `GET`/`POST` nest under the
member route. An inline panel in the bookings list was rejected — it crowds the list,
every row needs its own poll state, and it diverges from the "detail page" the spec
describes.

### API surface, authz, polling

- **Hirer-only, confirmed-only.** The Booking's `hirer` sees the panel; the Owner
  never does. Only `status == "confirmed"` (plus coverage) offers it.
- `GET .../trip_plan` → `{ status, interests, itinerary, error }`, or a "none" state
  → the frontend shows the interests form. Frontend polls every ~2.5s while
  `pending/processing`, stops on `completed/failed`, and renders an existing
  `completed` plan on load (the row is durable).
- **Error mapping** reuses ADR-0009: `InputError` → 400, `ApiError` → 503,
  `OutputError` → 500, rate limit → 429; plus **409** in-flight, **422** uncovered/
  invalid state, **403** non-Hirer, **404** missing/invisible Booking.
- `user: booking.hirer` is passed into `Ai::TripPlanner` so the `ai_requests` row is
  attributed to the Hirer.

### Evaluation framework: deferred

Consistent with ADR-0012 and the AI brief: TDD specs now (stub Claude and Ollama,
assert retrieval/region resolution, the coverage gate, the guards, the state
machine, schema validation), quality-regression evals later at the cross-cutting
build step.

## Deployment

The corpus is committed as markdown but the `knowledge_chunks` (embeddings) are
**not** — they are generated by `bin/rails knowledge:embed`, which reads every
Region's doc, chunks on `##` headings, and embeds via Ollama. The task is idempotent
on `content_hash`, so it is **safe to re-run on every deploy** (mirrors
`embeddings:backfill` from ADR-0011).

**A fresh environment starts with zero chunks.** Until `knowledge:embed` runs there,
every Region reports `has_corpus? == false`, so the coverage gate hides the panel on
*all* bookings and the feature appears absent. There is no automatic deploy hook yet
(same gap as `embeddings:backfill`), so this must be run manually — or wired into a
release step — after the first deploy and after any change to `app/knowledge/`.

## Alternatives Considered

**Hand-authored real corpus** — rejected for scope; the RAG pipeline is the lesson,
not curating travel data. Synthetic content is contained by the disclaimer + coverage
gate.

**Whole-doc injection instead of vector retrieval** — rejected; drops the headline
RAG lesson.

**State-level regions / no region pre-filter** — rejected; sub-state regions yield
better itineraries, and no pre-filter risks cross-region bleed (a VIC Booking
retrieving QLD chunks).

**`Region` as a DB table** — rejected; static reference data never edited at runtime;
a manifest is the single source of truth without a CRUD-less table to seed and sync.

**Static `town → region` hash at query time** — rejected; string lookup off to the
side that silently misses new towns. Resolved into a real `region` column instead.

**Append-only trip-plan history** — rejected as unneeded scope; one regenerable row
per Booking.

**Markdown-blob output** — rejected; no per-day structure to render or validate.

**Plan every day / default Sidekiq retries** — both rejected on cost/predictability
(unbounded tokens; up to 25 paid retries).

**Inline panel in the bookings list** — rejected; add a real booking-detail page
instead.
